Welcome to BiTranspiler

What does this do?

BiTranspiler translates code between C++17 and 32-bit x86 MASM Assembly — in both directions. It preserves your variables, loops, branches, and function frames through a shared AST engine.

C++ Code ⇄ AST Engine ⇄ MASM Assembly

C++ Writing Guidelines

Keep your C++ code within these constraints for clean compilation:

  • Entry Point: Always declare int main() { ... }.
  • Variables: Declared as local int or char before usage.
  • Conditionals: Supports nested if and if-else structures.
  • Loops: Write standard while loops or basic for loops.
  • Exits & Output: Returns display as the Exit Code. Standard output streams (cout or printf) print directly to the run log pane during C++ execution.

x86 MASM Writing Guidelines

Ensure your assembly adheres to this layout for clean recovery to C++:

  • Header Directives: Include .386, .model flat, stdcall, and ExitProcess PROTO at the top.
  • Data Segment: Define global variables inside .data using DD (for int) or DB (for char).
  • Procedures: Wrap statements in procedure frames (e.g. main PROC ... main ENDP). Use LOCAL var:DWORD for local scopes.
  • Exiting: Terminate execution with invoke ExitProcess, eax.

Reference Guide

Supported Features

  • Primitive Variable Types: Translates C++ local int and char variable declarations to MASM DD and DB directives.
  • Arithmetic Expressions: Fully recovers operations for addition (+), subtraction (-), multiplication (*), and division (/). Handles CDQ and IDIV assembly instructions recursively.
  • Control Flow Recovery: Translates flat label jumps back into structured C++ if, if-else, while loops, and standard for loops.
  • Procedures & Scopes: Supports multiple function definitions (PROC) targeting stack frame parameter allocations and local variables (LOCAL).
  • Calling Conventions: Implements the Stdcall calling convention for x86 architecture, returning outputs in EAX and handling exits using ExitProcess.

Out of Scope

  • Pointers, references, and dynamic memory allocations (e.g. new, delete, heap usage).
  • High-level C++ standard library structures (e.g. std::vector, std::string).
  • Object-Oriented Programming constructs (e.g. classes, inheritance, objects).

Q1: How does the transpiler recover structured blocks (like if-conditions or loops) from raw assembly JMPs?

Answer: The transpiler parses the MASM stream to identify JMP patterns. A backward JMP to a previous label marks a loop boundary, which gets parsed into a C++ while or for loop. Branching targets from conditional JMPs (like JE, JNE, JG, JL) denote then and else blocks. These blocks are recovered and mapped directly to C++ IfNode or WhileNode syntax elements in the AST.

Q2: Why do registers (like eax or ebx) sometimes appear as local variables in the recovered C++ code?

Answer: In order to guarantee compile-readiness, any register operations that cannot be safely collapsed into general C++ operators by the peephole optimizer are mapped directly to local C++ variables (e.g., int eax = 0;). This preserves register assignment logic. Whenever possible, however, standard instruction sequences (like mov eax, a; add eax, b; mov sum, eax) are simplified to high-level math statements (like sum = a + b;).

Q3: What assembly architecture and convention are used?

Answer: It targets 32-bit x86 MASM Assembly. The targeted calling convention is Stdcall, where function arguments are pushed onto the stack right-to-left, and returning integer values are placed inside the EAX register.

Q4: Why does Assembly use ExitProcess instead of returning?

Answer: ExitProcess is a Win32 API function that terminates the execution of a process. In standard MASM, this functions as the final exit mechanism for the main execution frame. The recovery engine detects invoke ExitProcess, eax at the end of the entry procedure and correctly translates it back to a standard C++ return eax; inside main().

Q5: How is bidirectional translation achieved under the hood?

Answer: The system shares a common Abstract Syntax Tree (AST) representation. The C++ parser maps C++ tokens into AST nodes, which the Assembly code-generator prints. Conversely, the Assembly parser maps MASM instructions into the same AST nodes, allowing the C++ printer to emit clean C++. This shared AST ensures logical alignment between both environments.

Q6: How can I see the execution output of my code?

Answer: There are two main ways to see execution outputs:
1. Exit Code (Recommended for Transpilation): In both MASM and C++, return your final result from the main procedure/function (which corresponds to invoke ExitProcess, eax in Assembly). When run, this value is printed as the Exit Code in the run output pane.
2. Console Printing (C++ Runs): You can write standard C++ output statements like cout << sum << endl; or printf("%d\n", sum); in the C++ editor and click Run C++. The console output (stdout) will print directly into the run output pane.
